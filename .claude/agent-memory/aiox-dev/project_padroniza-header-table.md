The `feat/padroniza-header-table` work migrates each domain list page (clientes, corretores, imobiliárias, unidades, comissões, vendas, membros, contratos) from the old `useQuery` + `<Card>` + `agency-pagination.tsx`-style offset pagination to the project's `DataTableInfinite` + `PageHeader` (fill-height) base.

**Why:** standardize the table/header language app-wide; the design source of truth is `docs/superpowers/specs/2026-06-11-row-language-column-maps.md` (§1 = cell-recipe principles, §2.x = per-domain column maps, §3 = sanity-check decisions). It is the design-first artifact that satisfies the `ux-gate` for Phase C.

**How to apply:** each migration (C1–C8) is a mechanical mirror of the **corretores** and **clientes** exemplars. Per domain, produce 4 edits: `use-<domain>-table.ts` (search-only via `useInfiniteTable`, 300ms debounce, `pageSize=20`), `<domain>-columns.tsx`, `<domain>-table.tsx`, `pages/<domain>/+Page.tsx`. Delete the orphaned `<domain>-pagination.tsx` with `git rm` in the same task. Add/rewrite the table smoke test and the hook test (the old hook test asserts the dead `pagination` API and must be rewritten to mirror `use-brokers-table.test.ts`). Per spec §3, only clientes/unidades/membros sort — corretores, imobiliárias, comissões, vendas, contratos are search-only (don't invent sort even when the endpoint exposes `sort_by`, per Constitution Art. IV). See [[dev-exemplar-mirroring-tables]].

**C7 (Membros) is the exception — `DataTable` base, NOT infinite.** Members lives in the `SettingsLayout` (Card-wrapped section, not a fill-height page), so per spec §2.7 it uses the **base `DataTable`** + keeps its existing offset `MembersPagination` (the hook stays on `useQuery` + `page`/`page_size`, not `useInfiniteTable`). The base `DataTable`'s `min-h-0 flex-1 overflow-auto` is inert without a flex-fill parent, so it sizes to content (≤10 rows) — that's the intended finite behavior. Members already had server sort (`sort_by`, default `created_at:desc`), so wire `meta={{ sort, onSort }}` + `SortableHeader` for `full_name` and `created_at` (one of only 3 sorting domains per §3). Avatar âncora: compose `[<Avatar/> <PrimaryCell/>]` inline in the `full_name` cell (spec §2.7 option a — don't add a `leading?` prop to `PrimaryCell`). `created_at` uses the canonical `DateCell` (`format(parseISO,'dd/MM/yyyy')`), not the legacy `toLocaleDateString`. Error state moves into `DataTable` (`isError`/`onRetry`), so the section drops its manual retry block. C7 done at commit 2d25971. `getRowId = String(member.id)` (id is a uuid). The members e2e spec only checks tabs/headings (not table internals), so it survives the migration.

**Summary blocks are not uniform.** Financeiro/installments embeds its `summary` in the list response (`page.response` → `useInfiniteTable.firstResponse`). **Comissões does NOT** — `/api/v1/commissions` returns only `{items,total}`; the totals live in a separate non-paginated endpoint `/api/v1/commissions/summary` (verified in api-client schema.d.ts 1538). So C5 keeps the list on `useInfiniteTable` but the summary stays a parallel `useQuery` keyed on the same `filterParams`, fed to `commission-summary-cards` above the table (`shrink-0`). Before assuming `firstResponse` carries the summary for any domain, check whether the list operation's 200 response actually contains a summary field. C5 done at commit 3941518 (Data column header is "Data" but field is `sale_closed_at`; sale link uses `formatId(sale_id)` → `#00042` mono, since `sale_id` is a numeric PK not a UUID).
