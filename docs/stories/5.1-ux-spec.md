# UX Spec — Story 5.1: Empreendimento — campos jurídicos, registrais e de construção

## Objetivo da tela / componente

Estender o formulário compartilhado `ProjectForm` (usado em criação e edição de empreendimentos) com:

1. **1 campo novo** no card "Informações Básicas" existente — `total_area`
2. **3 novos cards** ao final do formulário, antes dos botões de ação:
   - "Pessoa Jurídica (SPE)" — dados da SPE do empreendimento
   - "Registros e Alvarás" — dados registrais e documentos de obra
   - "Inscrições" — inscrições estadual e municipal

Todos os campos são **opcionais** — o formulário já pode ser submetido sem preenchê-los. A experiência deve ser **aditiva**: o formulário existente não muda de comportamento; os novos cards são seções complementares de preenchimento opcional.

---

## Fluxo principal do usuário

### Cenário A — Criação de empreendimento (rota `/empreendimentos/novo`)

```
Usuário acessa "Novo Empreendimento"
  → Preenche campos obrigatórios (nome, status, CEP, cidade, UF, logradouro, número, bairro)
  → [Opcional] Preenche "Área Total" no card Informações Básicas
  → [Opcional] Desce até "Pessoa Jurídica (SPE)" → preenche CNPJ com máscara aplicada automaticamente
  → [Opcional] Preenche Razão Social e Nome Fantasia
  → [Opcional] Desce até "Registros e Alvarás" → preenche campos relevantes
  → [Opcional] Desce até "Inscrições" → preenche IE e/ou IM
  → Clica "Cadastrar"
  → Loading spinner no botão
  → Toast de sucesso + redirecionamento
```

### Cenário B — Edição de empreendimento (rota `/empreendimentos/:id/editar`)

```
Usuário acessa "Editar Empreendimento"
  → Form carrega com todos os campos preenchidos (existentes + novos)
  → Campos novos mostram valores já salvos (ou vazios se nunca preenchidos)
  → Usuário edita o que precisar
  → Clica "Atualizar"
  → Loading spinner no botão
  → Toast de sucesso
```

### Cenário C — CNPJ duplicado (409)

```
Usuário preenche CNPJ já cadastrado para outro empreendimento
  → Clica "Cadastrar" / "Atualizar"
  → API retorna 409
  → handleApiError exibe toast.error com mensagem do backend
  → Formulário permanece preenchido (sem reset)
  → Usuário corrige o CNPJ e tenta novamente
```

### Cenário D — CNO inválido (validação inline)

```
Usuário digita CNO com != 12 dígitos
  → Ao sair do campo (onBlur) ou ao submeter
  → FormMessage exibe "CNO deve ter exatamente 12 dígitos"
  → Campo fica com borda vermelha (estado de erro padrão shadcn)
  → Submit bloqueado enquanto campo inválido
```

## Estados da UI

### Card "Informações Básicas" — campo `total_area`

| Estado | Comportamento |
|--------|--------------|

| Estado | Campo | Comportamento |
|--------|-------|--------------|

| Vazio | `legal_name` | Placeholder vazio; FormDescription: "Quando vazio, herda os dados da organização" |

### Card "Registros e Alvarás"

| Estado | Campo | Comportamento |
|--------|-------|--------------|
| Vazio | `cno` | Placeholder "000000000000"; FormDescription: "Exatamente 12 dígitos" |

| CNO com 12 dígitos | `cno` | Válido; submit liberado para este campo |
| Demais campos | todos | Inputs de texto livre; sem validação de formato |

### Card "Inscrições"

| Estado | Comportamento |
|--------|--------------|
| Vazio | Inputs de texto livre; sem validação especial |
| Preenchido | Valor exibido normalmente |

### Estados globais do formulário

| Estado | Comportamento |
|--------|--------------|
| `isSubmitting = true` | Botão "Cadastrar"/"Atualizar" mostra `<Loader2 animate-spin>` + texto de loading; botão desabilitado |

| Campos obrigatórios ausentes | Submit bloqueado; FormMessage nos campos com erro |
| Todos obrigatórios preenchidos + opcionais inválidos | Submit bloqueado pelos campos inválidos (CNPJ, CNO) |

### Máscara CNPJ

- Componente: `<CNPJInput>` de `src/components/ui/document-input.tsx`
- Máscara aplicada ao digitar: `XX.XXX.XXX/XXXX-XX`
- Não exige input numérico puro — a máscara formata automaticamente
- `maxLength=18` já definido internamente no componente
- Placeholder padrão `"00.000.000/0000-00"` já definido internamente

### Campo CNO

- `inputMode="numeric"` — abre teclado numérico em mobile
- `maxLength={12}` — limita a 12 caracteres (apenas dígitos esperados)
- `placeholder="000000000000"`
- `<FormDescription>Exatamente 12 dígitos</FormDescription>` abaixo do campo
- Validação Zod: `!val || val.replace(/\D/g, '').length === 12`

### FormDescription nos campos SPE

Os campos `cnpj` e `legal_name` têm `<FormDescription>` com texto:  
**"Quando vazio, herda os dados da organização"**

### Comportamento de campos já cadastrados (edição)

- Campos novos com valor `null` ou `undefined` no backend → `defaultValue: ''` → campo aparece vazio
- Campos já preenchidos → `defaultValue: initialData?.campo || ''` → valor exibido normalmente
- Nenhum campo novo bloqueia a edição de empreendimentos antigos

### Validação CNPJ — campos opcionais

- Campo **vazio** (`''` ou `null`) → **válido** (campo opcional)
- Campo **preenchido com CNPJ inválido** → `FormMessage: "CNPJ inválido"`
- Campo **preenchido com CNPJ válido** → sem erro
- Validação só dispara quando o campo tem valor (`if (!val) return true`)

### Tratamento de erro 409 (CNPJ duplicado por tenant)

- `handleApiError` do `onError` callback já extrai `body.message` do backend
- Toast `toast.error(mensagem)` exibido em `bottom-center` com `richColors`
- Formulário **não reseta** — usuário pode corrigir e retentar
- Nenhuma lógica especial além do padrão Pattern A já em uso no form

## Referências

### Design system

- **Tokens:** OKLch color space, Inter font, 8px grid spacing (`.design-engineer/system.md`)
- **Card pattern:** `<Card><CardHeader><CardTitle> + <CardContent className="space-y-4">`
- **FormField pattern:** `<FormField> → <FormItem> → <FormLabel> + <FormControl> + <FormDescription?> + <FormMessage>`
- **Grid:** `grid gap-4 sm:grid-cols-12` + `sm:col-span-{N}` (padrão já estabelecido no form)

### Componentes shadcn/ui reutilizados

| Componente | Origem | Uso |
|-----------|--------|-----|
| `<Card>`, `<CardHeader>`, `<CardTitle>`, `<CardContent>` | shadcn/ui | Estrutura dos novos cards |
| `<FormField>`, `<FormItem>`, `<FormLabel>`, `<FormControl>`, `<FormDescription>`, `<FormMessage>` | shadcn/ui | Todos os campos |
| `<Input>` | shadcn/ui | Todos os campos texto simples |
| `<CNPJInput>` | `src/components/ui/document-input.tsx` | Campo `cnpj` |

### Padrão de referência — módulo customers

O CNPJ input no módulo customers (`src/components/customers/customer-form.tsx`) é o modelo exato a seguir — mesma estrutura `FormField + CNPJInput + FormDescription`. A única diferença é a guard `if (!val) return true` no `.refine()` para tornar o campo opcional.
