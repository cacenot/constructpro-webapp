# UX Spec — Story 4.7: Índice de Correção por Grupo de Parcelas

## Objetivo da tela / componente

Adicionar ao Step 3 ("Pagamento") do wizard de nova proposta um **toggle de modo de indexação**:

- **Modo A — índice único (default):** um seletor global `index_type_code` aplica o mesmo índice para todos os grupos de parcelas. Comportamento idêntico ao atual.
- **Modo B — índice por grupo:** o seletor global desaparece; cada grupo no `<InstallmentScheduleBuilder>` exibe seu próprio campo "Índice de Correção". Permite ex: parcelas regulares em IGPM + entrega das chaves em INCC.

A mesma lógica de toggle se aplica ao `sale-edit-form.tsx`.

---

## Fluxo principal do usuário

```
[Step 3 — Pagamento]
    │
    ▼
Toggle "Usar o mesmo índice para toda a proposta" → ON (default)
  ├── Seletor global "Índice de Correção *" visível acima do builder
  ├── Usuário seleciona índice global (ex: IGPM)
  ├── Configura grupos no InstallmentScheduleBuilder normalmente
  └── Clica "Salvar Proposta" → payload: { index_type_code: "IGPM", schedules: [...] }

Toggle alterado para OFF
  ├── Seletor global desaparece
  ├── Cada grupo no builder exibe seletor "Índice de Correção *" próprio
  │   ├── Grupos já adicionados: campo pré-preenchido com o valor do índice global anterior
  │   └── Novos grupos adicionados: campo vazio (obrigatório)
  ├── Usuário pode atribuir índices diferentes por grupo
  └── Clica "Salvar Proposta" → payload: { schedules: [{ index_type_code: "IGPM", ... }, { index_type_code: "INCC", ... }] }

Toggle alternado de OFF → ON
  ├── Seletor global reaparece
  ├── Valor pré-preenchido com o índice do primeiro grupo (ou vazio se nenhum grupo)
  └── Seletores por grupo desaparecem do builder
```

## Comportamento de Transição de Toggle

### Ao alternar de ON → OFF (global → por grupo)

1. Para cada grupo já adicionado no builder:
   - `form.setValue(`installment_schedules.${i}.index_type_code`, globalIndexTypeCode)`
   - Onde `globalIndexTypeCode = form.getValues('index_type_code')` capturado no momento da alternância
2. Se `globalIndexTypeCode` estiver vazio/undefined: campos por grupo ficam vazios (obrigatório — validação Zod bloqueia submit)
3. O seletor global desaparece

### Ao alternar de OFF → ON (por grupo → global)

1. `form.setValue('index_type_code', schedules[0]?.index_type_code ?? '')`
   - Usa o índice do **primeiro grupo** como valor inicial do seletor global
   - Se nenhum grupo existe ou todos têm `index_type_code` vazio: seletor global fica vazio
2. Os seletores individuais desaparecem do builder (mas os valores continuam no `form` state — não há `setValue` para limpá-los, pois não são incluídos no payload quando toggle ON)

> **Razão:** Evitar perda de dado se o usuário alternar acidentalmente. Os valores por grupo ficam "ocultos" no estado mas não são enviados quando toggle ON.

## Layout Mobile (< lg)

- Toggle ocupa largura completa (`w-full` implícito por ser block element)
- `<Switch>` e label em `flex items-center gap-3` — sem mudança
- Seletor global: `w-full` (sem grid `sm:grid-cols-12` — colapsa para col única)
- Seletor por grupo: idem — largura completa em mobile, sem grid
- Comportamento de transição: idêntico ao desktop

**Não há comportamento diferenciado no mobile** além do colapso normal do grid responsivo.

## sale-edit-form.tsx — Inicialização do Toggle

Ao carregar a proposta existente, determinar o estado inicial do toggle:

```
Se algum schedule tem index_type_code preenchido individualmente
  → toggle OFF (sameIndexForAll = false)
  → seletor global oculto
  → cada grupo mostra seu índice carregado do banco

Se nenhum schedule tem index_type_code individual (campo ausente ou null em todos)
  → toggle ON (sameIndexForAll = true) — default
  → seletor global carregado com sale.index_type_code
```

**Lógica de detecção:**
```ts
const hasPerGroupIndex = sale.installment_schedules?.some(
  (s) => s.index_type_code != null && s.index_type_code !== ''
)
const [sameIndexForAll, setSameIndexForAll] = useState(!hasPerGroupIndex)
```

## Decisão Técnica: Propagação do `sameIndexForAll`

**Abordagem adotada: Opção A — elevar `sameIndexForAll` para `sale-form.tsx`**

O estado `sameIndexForAll` vive em `sale-form.tsx` como `useState<boolean>(true)`. É passado como prop para `sale-form-step3.tsx` (com setter) e para `InstallmentScheduleBuilder` (somente leitura).

**Razão:** `sale-form.tsx` é quem constrói o payload antes do submit. Para montar o payload correto (global vs. por grupo), precisa do valor de `sameIndexForAll` diretamente, sem callback intermediário.

```ts
// Em sale-form.tsx
const [sameIndexForAll, setSameIndexForAll] = useState<boolean>(true)

// Passa para Step 3:
<SaleFormStep3
  ...
  sameIndexForAll={sameIndexForAll}
  onToggleChange={setSameIndexForAll}
/>

// Step 3 passa para o builder:
<InstallmentScheduleBuilder
  ...
  sameIndexForAll={sameIndexForAll}
  indexTypes={indexTypes}
/>
```

## Estados da UI

### Loading

| Elemento | Estado | Comportamento |
|----------|--------|---------------|
| Seletor global | `indexTypesLoading` | `disabled` + placeholder "Carregando..." |
| Seletor por grupo | idem | `disabled` + placeholder "Carregando..." |
| Toggle | — | Sempre habilitado (não depende de API) |

### Empty

| Elemento | Condição | Comportamento |
|----------|----------|---------------|
| Seletor global (toggle ON) | Sem opções em `indexTypes` | `disabled`, placeholder "Nenhum índice disponível" |
| Seletor por grupo (toggle OFF) | idem | idem |
| Nenhum grupo adicionado (toggle OFF) | `fields.length === 0` | Toggle OFF visível; builder mostra estado vazio normal; nenhum seletor por grupo renderizado |

### Error

| Elemento | Condição | Comportamento |
|----------|----------|---------------|
| Seletor global | submit com campo vazio (toggle ON) | `<FormMessage>` padrão shadcn: "Índice de correção é obrigatório" |
| Seletor por grupo | submit com campo vazio em algum grupo (toggle OFF) | `<FormMessage>` abaixo do select do grupo: "Índice de correção é obrigatório" |
| Toggle | — | Sem estado de erro |

### Success

| Elemento | Condição | Comportamento |
|----------|----------|---------------|
| Submit (toggle ON) | Proposta criada | `toast.success` + redirect (comportamento existente) |
| Submit (toggle OFF) | Proposta criada com índices por grupo | `toast.success` + redirect (mesmo comportamento) |

## Comportamentos Esperados

### Toggle

| Ação | Resultado visual |
|------|-----------------|
| Estado inicial | Toggle ON, seletor global visível (estado atual preservado) |
| Clicar toggle (ON → OFF) | Seletor global desaparece com transição CSS padrão; seletores por grupo aparecem em cada grupo já adicionado |
| Clicar toggle (OFF → ON) | Seletores por grupo desaparecem; seletor global reaparece com valor do primeiro grupo |
| Adicionar novo grupo com toggle OFF | Grupo renderiza imediatamente com seletor de índice vazio (obrigatório) |
| Remover todos os grupos com toggle OFF | Builder fica vazio; nenhum seletor por grupo visível |

### Validação Zod

| Modo | Validação |
|------|-----------|
| Toggle ON | `saleFormSchema.index_type_code` obrigatório; `installmentScheduleSchema.index_type_code` ignorado no payload |
| Toggle OFF | `saleFormSchema.index_type_code` não enviado no payload; `superRefine` em cada schedule exige `index_type_code` preenchido |

### Sem regressão

Propostas existentes com toggle ON (comportamento atual) devem ter **payload idêntico** ao pré-story-4.7. Nenhum campo existente alterado no modo padrão.
