# UX Spec — Story 4.4: Montagem de Parcelas

## Objetivo da tela / componente

Extrair e redesenhar a seção de parcelas de `sale-form-step2.tsx` em um componente autônomo `<InstallmentScheduleBuilder>` que:

- Organiza todos os schedules em **5 grupos visuais** com subtotais em tempo real
- Expande o dropdown "Adicionar Parcelas" para cobrir todos os 5 tipos (`entry`, `regular`, `balloon`, `key_delivery`, `extra`)
- Refatora o sidebar "Detalhamento" de `sale-form.tsx` para exibir subtotais por grupo em vez de linha por linha
- É reutilizável em `sale-edit-form.tsx` sem duplicação de lógica

**Escopo visual:** Step 2 do wizard de Nova Proposta + Form de Edição de Proposta

---

## Fluxo principal do usuário

### Cenário A — Criação de proposta

1. Usuário chega ao Step 2 (financeiro) após selecionar empreendimento + unidade no Step 1
2. O builder é renderizado com o grupo **Entradas** já visível (1 entrada default)
3. Usuário preenche a(s) entrada(s) — subtotal do grupo "Entrada" atualiza em tempo real no builder e no sidebar
4. Usuário clica em "Adicionar Parcelas" → dropdown com todos os tipos aparece
5. Usuário seleciona tipo (ex: "Mensais") → grupo "Regular" aparece no builder com card pré-preenchido
6. Usuário preenche campos do schedule regular — subtotal do grupo "Regular" atualiza
7. Opcionalmente usuário adiciona Balão, Entrega das Chaves, Extra
8. Sidebar exibe subtotais por grupo + total geral em tempo real
9. Usuário submete a proposta

### Cenário B — Edição de proposta existente

1. Usuário acessa a tela de edição (`sale-edit-form.tsx`)
2. Builder renderiza todos os schedules existentes organizados por grupo (carregados da API)
3. Usuário edita valores/datas → subtotais atualizam em tempo real
4. Usuário pode adicionar novos schedules ou remover existentes
5. Usuário salva

### Grupos visuais

| Grupo | `kind` | Label PT-BR | Sempre visível? |
|-------|--------|-------------|-----------------|
| Entradas | `entry` | Entradas | **Sim** (default com 1 entrada) |
| Regular | `regular` | Parcelas Regulares | Não — aparece ao adicionar |
| Balão / Reforço | `balloon` | Balões / Reforços | Não — aparece ao adicionar |
| Entrega das Chaves | `key_delivery` | Entrega das Chaves | Não — aparece ao adicionar |
| Extras | `extra` | Extras | Não — aparece ao adicionar |

**Decisão UX:** Grupos sem schedules ficam **completamente ocultos** no builder. O dropdown "Adicionar Parcelas" é o único ponto de entrada para todos os tipos. Grupos aparecem quando o primeiro schedule daquele tipo é adicionado.

### Header de grupo

Cada grupo visível exibe um header com:
- Label em PT-BR (bold, `text-sm font-medium`)
- Subtotal do grupo alinhado à direita (`tabular-nums`, atualiza em tempo real)
- Subtotal = `Σ (quantity × amount)` de todos schedules do grupo
- Formato: `R$ X.XXX,00`

```
Entradas                              R$ 50.000,00
```

## Sidebar "Resumo Financeiro" — seção Detalhamento (refatoração)

### Comportamento atual (a ser substituído)

Lista cada schedule individualmente com label + subtotal, pulando schedules com subtotal zero.

### Comportamento novo (Story 4.4)

Agrupa por `kind`, exibe subtotal por grupo. Oculta grupos com subtotal zero.

```
Detalhamento

Entradas                          R$ 50.000,00
Parcelas Regulares               R$ 120.000,00
Balões / Reforços                 R$ 20.000,00
──────────────────────────────────────────────
Valor Financiado                 R$ 190.000,00
```

**Regras:**
- Label de grupo = `INSTALLMENT_KIND_LABELS[kind]`, mas com aliases visuais:
  - `entry` → "Entradas"
  - `regular` → "Parcelas Regulares"
  - `balloon` → "Balões / Reforços"
  - `key_delivery` → "Entrega das Chaves"
  - `extra` → "Extras"
- Subtotal de grupo = `Σ (quantity × amount)` de todos schedules do grupo
- Grupos com subtotal = 0 são **omitidos** do sidebar
- Ordem de exibição: Entradas → Regulares → Balões → Chaves → Extras
- Total geral "Valor Financiado" = Σ de todos os grupos (inalterado)
- Formatação: `tabular-nums`, formato BRL via `formatCentsToDisplay`

## Comportamentos esperados

### Subtotais em tempo real

- Subtotal de cada grupo recalcula ao alterar `quantity` ou `amount` de qualquer schedule do grupo
- Atualização via `useWatch` — sem debounce (mesma abordagem do sidebar atual)
- Subtotal do sidebar também atualiza em tempo real

### Adição de schedule

- Clicar em item do dropdown → `append()` com defaults corretos → grupo do tipo aparece (se era o primeiro do tipo)
- Foco automático no primeiro campo `quantity` do card recém-adicionado (via `quantityInputRefs`, padrão existente)
- Scroll suave para o card adicionado (`scrollIntoView({ behavior: 'smooth', block: 'nearest' })`, padrão existente)

### Remoção de schedule

- Botão `Trash2` com tooltip "Remover parcela"
- Chama `remove(index)` imediatamente (sem confirmação)
- Se era o último schedule do grupo → grupo desaparece do builder

### Reuso em sale-edit-form.tsx

- Mesma renderização visual que no form de criação
- Schedules pre-populados via `fields` do `useFieldArray`
- Props idênticas: `form`, `fields`, `append`, `remove`, `watchedSchedules`
- Funções `addXxxSchedule` ficam **dentro do componente** — edit form não precisa passar como props

## Decisões de UX documentadas

| Decisão | Escolha | Razão |
|---------|---------|-------|
| Grupos vazios no builder | Ocultos completamente | Reduz ruído visual; dropdown é o ponto de entrada explícito |
| Grupo "Entradas" | Sempre visível | É o grupo default com 1 schedule pré-adicionado |
| key_delivery quantity | Editável (default=1) | Pode haver mais de uma data de entrega em alguns contratos |
| extra recurrence_type | Fixo em `monthly` nos defaults | Simplifica UI; casos avançados via API |
| Ordem dos grupos no builder | Entradas → Regulares → Balões → Chaves → Extras | Ordem cronológica natural do contrato |
| Confirmação ao remover | Não | Consistência com comportamento atual |
| Sidebar Detalhamento | Por grupo, grupos zero ocultos | Visão gerencial mais limpa que linha por linha |

## Restrições técnicas para @dev

1. **Props interface do builder:**
   ```ts
   interface InstallmentScheduleBuilderProps {
     form: UseFormReturn<SaleFormData>
     fields: FieldArrayWithId<SaleFormData, 'installment_schedules'>[]
     append: UseFieldArrayAppend<SaleFormData, 'installment_schedules'>
     remove: UseFieldArrayRemove
     watchedSchedules: SaleFormData['installment_schedules'] | undefined
   }
   ```

2. **Funções de adição** ficam **dentro** do componente (não como props)

3. **Helper `groupSchedulesByKind`** retorna `Map<InstallmentKind, number[]>` (índices no array `fields`)

4. **Subtotal de grupo** = `fields.filter(kind).reduce((sum, idx) => sum + qty[idx] * amount[idx], 0)`

5. **Zod gaps** (Task 3 da story): adicionar validação para `key_delivery` e `extra` no `superRefine` de `sale.schema.ts`

6. **Sidebar** usa os mesmos `watchedSchedules` para calcular subtotais — não precisa de nova prop

7. **Sem nova dependência externa** — tudo usa componentes e hooks já presentes no projeto
