# UX Spec — Story 4.1: Wizard Step Empreendimento × Unidade

## Objetivo da tela / componente

Transformar o formulário de Nova Proposta em um **wizard de 2 passos**:

- **Step 1 — Empreendimento e Unidade:** Usuário seleciona o empreendimento via `<ProjectAutocomplete>`, depois a unidade filtrada por `project_id` via `<UnitAutocomplete projectId=…>`.
- **Step 2 — Dados Financeiros:** Exibe a seleção feita no Step 1 como chip read-only no topo, seguido de todos os campos financeiros existentes (Dados da Venda sem o campo Unidade, Comissão, Pagamento) e o painel de resumo lateral.

**Motivação da UX:** O formulário atual (~757 linhas) exibe unidades de todos os empreendimentos sem filtro, criando ruído na seleção. O wizard cria um fluxo guiado e progressivo: primeiro define "onde" (empreendimento + unidade), depois "como" (condições financeiras). A unidade disponível fica contextualizada no projeto antes de qualquer informação financeira.

**Escopo:** Brownfield enhancement em `sale-form.tsx`. O wizard se aplica apenas ao formulário de **criação**; `sale-edit-form.tsx` não é afetado.

## Fluxo principal do usuário

```
Usuário acessa "Nova Proposta"
│
├─ STEP 1 — Empreendimento e Unidade
│   │
│   ├─ Vê ProjectAutocomplete habilitado
│   │   UnitAutocomplete desabilitado (placeholder: "Selecione um empreendimento primeiro")
│   │   Botão "Próximo" desabilitado
│   │
│   ├─ Busca e seleciona empreendimento
│   │   └─ UnitAutocomplete habilita, dispara query com project_id
│   │       Botão "Próximo" ainda desabilitado
│   │
│   ├─ Busca e seleciona unidade
│   │   └─ Botão "Próximo" habilita
│   │
│   ├─ (Opcional) Troca empreendimento
│   │   └─ UnitAutocomplete reseta e desabilita novamente
│   │       Botão "Próximo" desabilita
│   │
│   └─ Clica "Próximo"
│       └─ Transição para Step 2
│
├─ STEP 2 — Dados Financeiros
│   │
│   ├─ Chip read-only no topo mostra: "Unidade selecionada: {nome} — {empreendimento}"
│   │
│   ├─ Preenche campos financeiros (Cliente, Índice, Comissão, Parcelas)
│   │
│   ├─ (Opcional) Clica "Voltar"
│   │   └─ Retorna ao Step 1 com seleções preservadas
│   │
│   └─ Clica "Criar Proposta"
│       └─ Submit existente (sem alteração)
│
└─ Sucesso / erro de submit → toast (comportamento atual preservado)
```

## Comportamentos esperados

### B1 — Indicador de progresso (step indicator)

Componente inline simples, sem biblioteca externa:

```
● ──────────── ○
```

```tsx
// Markup semântico usando ol/li com aria-current
<ol className="flex items-center gap-4">
  <li aria-current={step === 1 ? 'step' : undefined}>
    <div className={cn(
      'flex h-8 w-8 items-center justify-center rounded-full text-sm font-medium',
      step === 1
        ? 'bg-primary text-primary-foreground'        // ativo
        : 'bg-primary text-primary-foreground'         // concluído (mesmo estilo + ✓)
    )}>
      {step > 1 ? <Check className="size-4" /> : '1'}
    </div>
    <span className="ml-2 text-sm">Empreendimento e Unidade</span>
  </li>
  <li className="h-px flex-1 bg-border" aria-hidden="true" />   {/* linha separadora */}
  <li aria-current={step === 2 ? 'step' : undefined}>
    <div className={cn(
      'flex h-8 w-8 items-center justify-center rounded-full text-sm font-medium',
      step === 2
        ? 'bg-primary text-primary-foreground'
        : 'border-2 border-muted-foreground text-muted-foreground'  // inativo
    )}>
      2
    </div>
    <span className={cn('ml-2 text-sm', step < 2 && 'text-muted-foreground')}>
      Financeiro
    </span>
  </li>
</ol>
```

- Sem animação de transição entre steps — mudança imediata (performance, sem framer-motion)
- `aria-current="step"` no `<li>` ativo para acessibilidade

### B2 — Habilitação condicional de UnitAutocomplete

```tsx
<UnitAutocomplete
  value={field.value}
  onChange={handleUnitChange}
  projectId={selectedProject?.id}         // undefined quando nenhum projeto selecionado
  disabled={!selectedProject}             // explicitamente disabled sem projeto
  placeholder={
    selectedProject
      ? 'Selecione uma unidade...'
      : 'Selecione um empreendimento primeiro'
  }
/>
```

- `disabled={!selectedProject}` garante que o campo não receba foco via teclado
- `aria-disabled="true"` deve ser propagado internamente em `UnitAutocomplete` quando `disabled`

### B3 — Reset ao trocar empreendimento

Ao `ProjectAutocomplete.onChange` disparar com projeto diferente do atual:

```ts
const handleProjectChange = (project: SelectedProject | null) => {
  if (project?.id !== selectedProject?.id) {
    // Reset unit apenas se projeto mudou (não no re-render)
    form.setValue('unit_id', undefined)
    setSelectedUnit(null)
  }
  setSelectedProject(project)
}
```

- Comparação por `id` para evitar reset em re-renders
- `form.setValue('unit_id', undefined)` — usa `undefined` (não `null`) para consistência com `defaultValues`

### B4 — Botão "Próximo" — condição de habilitação

```tsx
const canProceed = selectedProject !== null && form.watch('unit_id') != null

<Button
  type="button"
  disabled={!canProceed}
  onClick={() => setStep(2)}
>
  Próximo
  <ArrowRight className="ml-2 size-4" />
</Button>
```

- `type="button"` obrigatório — não deve submeter o formulário
- Ícone `ArrowRight` (lucide-react, já importado no projeto)

### B5 — Chip de seleção no Step 2 (read-only)

```
┌──────────────────────────────────────────────────────────────────┐
│ ✓  Residencial Parque das Flores                                 │
│    Apto 301 — Bloco B  ·  R$ 450.000,00          [Alterar]       │
└──────────────────────────────────────────────────────────────────┘
```

- Renderizado com `<Card>` + `bg-muted/50` para destaque sutil (não `bg-primary` — não é ação, é info)
- `<Check className="text-green-600">` para indicar concluído
- "Alterar" → `<Button variant="ghost" size="sm" onClick={() => setStep(1)}>Alterar</Button>`
- Preço formatado com `formatCentsToDisplay(selectedUnit.price_cents)` (utilitário já existente)

### B6 — Navegação entre steps

| Ação | Comportamento |
|------|---------------|
| Clicar "Próximo" | `setStep(2)` — sem limpar campos |
| Clicar "Voltar ao Step 1" | `setStep(1)` — sem limpar campos |
| Clicar "Alterar" no chip | `setStep(1)` — sem limpar campos |
| Clicar "← Voltar" no header | `onBack()` — sai do formulário inteiro (comportamento atual) |

**Importante:** "Voltar" no header do formulário (`onBack`) sai do fluxo de criação de proposta (comportamento atual preservado). É diferente de "Voltar ao Step 1".

### B7 — UnitAutocomplete: mensagem de lista vazia com projectId

Quando `projectId` definido e nenhuma unidade retornada:

```tsx
<CommandEmpty>
  {debouncedSearch
    ? 'Nenhuma unidade disponível encontrada.'
    : projectId
      ? 'Nenhuma unidade disponível neste empreendimento.'
      : 'Digite para buscar...'}
</CommandEmpty>
```

### B8 — Campo unit_id removido do Step 2 / Card "Dados da Venda"

O Card "Dados da Venda" no Step 2 fica com **apenas** `customer_id` e `index_type_code`. O campo `unit_id` do formulário é gerenciado exclusivamente pelo Step 1 via `handleUnitChange`.

## Referências

### Design system / tokens

- **Espaçamento:** `space-y-6` entre cards (padrão do `sale-form.tsx`) — manter consistência
- **Grid:** `grid grid-cols-1 lg:grid-cols-[1fr_340px]` — estrutura externa preservada no Step 2
- **Card container:** `<Card>` + `<CardHeader>` + `<CardContent className="space-y-4">` — padrão existente
- **Chip de seleção:** `<Card>` com `bg-muted/50 border-muted` — estilo informativo (não ação)
- **Cores:** `text-green-600` para ✓ do chip e step indicator concluído — consistente com feedback positivo
- **Step indicator círculos:** `h-8 w-8 rounded-full` — 8px grid; `bg-primary` ativo, `border-2 border-muted-foreground` inativo

### Fluxo de decomposição de `sale-form.tsx`

O `SaleForm` (orquestrador) mantém:
- `step` state
- `selectedProject` state
- `form` (react-hook-form) — compartilhado entre steps
- Renderiza `<SaleFormStep1>` ou `<SaleFormStep2>` conforme `step`

`SaleFormStep1` recebe:
- `form` (para `FormField unit_id`)
- `selectedProject` + `onProjectChange`
- `selectedUnit` + `onUnitChange`
- `onNext` callback

`SaleFormStep2` recebe:
- `form` (para todos os campos financeiros)
- `selectedUnit` (para chip read-only)
- `selectedProject` (para chip read-only)
- `onBack` callback
- `isSubmitting`
- Dados de queries (brokers, agencies, indexTypes)
