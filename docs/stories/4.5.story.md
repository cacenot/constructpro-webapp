## Story

**As a** corretor montando uma proposta,
**I want** ser avisado imediatamente quando adiciono parcelas em excesso num mesmo mês-calendário,
**so that** eu corrija o problema antes de tentar salvar — e entenda claramente o motivo quando o backend rejeitar a proposta com 422.

## Acceptance Criteria

1. A função `computeInstallmentsPerMonth(schedules)` é adicionada a `src/lib/installment-utils.ts`. Ela retorna um `Map<string, number>` onde a chave é `'YYYY-MM'` e o valor é o total de parcelas individuais que vencem naquele mês. Para schedules recorrentes expande todas as datas pelo `recurrence_type` + `start_date` + `quantity` + `recurrence_day` usando `addMonthsClamped` já existente; para `entry` usa `specific_date` multiplicado por `quantity`.

2. O `<InstallmentScheduleBuilder>` aceita a prop opcional `maxInstallmentsPerMonth?: number`. Quando informada e maior que 0, calcula os meses violados a partir de `watchedSchedules` e exibe, **abaixo do botão "Adicionar Parcelas"**, um aviso inline por mês violado:
   > ⚠️ **Agosto/2026**: 3 parcelas — limite é 2. Ajuste as datas ou reduza as quantidades.

   O aviso usa cor de alerta (`text-destructive` ou `text-warning`). Não bloqueia o submit — é apenas feedback visual em tempo real. Se não houver violações, nada é exibido.

3. Em `sale-form.tsx`, o hook `useTenantConfig()` (já existente em `src/hooks/use-tenant-config.ts`) é chamado para obter `max_installments_per_month`. O valor (`config?.max_installments_per_month ?? 2`) é repassado como prop `maxInstallmentsPerMonth` ao `<SaleFormStep2>`, que o passa ao `<InstallmentScheduleBuilder>`.

4. Na tela de **nova proposta**, ao submeter com violação de teto, o backend retorna 422 com estrutura:
   ```json
   {
     "error_code": <N>,
     "message": "...",
     "details": { "month": "2026-08", "count": 3, "cap": 2 }
   }
   ```
   O tratamento usa `handleApiError` padrão em `onError` da mutation (`pages/vendas/novo/+Page.tsx`). Se o backend já devolve `message` descritiva, isso é suficiente. Caso `message` seja genérica, o `catch` verifica `error_code` ou inspeciona `details` para formatar: `"Mês 08/2026: {count} parcelas excedem o limite de {cap}."`.

5. O campo `max_installments_per_month` é adicionado ao schema Zod `tenantConfigSchema` (`src/schemas/tenant-config.schema.ts`) como:
   ```ts
   max_installments_per_month: z.number().int('Deve ser inteiro').min(1).max(5)
   ```
   Tipo `TenantConfigFormData` é atualizado automaticamente via inferência.

6. `TenantConfigForm` (`src/components/configuracoes/tenant-config-form.tsx`) recebe um novo `<Card>` com título **"Parcelas por Mês"** e descrição "Máximo de parcelas com vencimento no mesmo mês-calendário". Dentro do card: um **slider numérico** (`<input type="range" min={1} max={5} step={1}>`) acompanhado de um display do valor atual (`{value} parcela{value > 1 ? 's' : ''}`) e legenda "Entre 1 e 5 (padrão: 2)". Segue o padrão `FormField` + `FormControl` + `FormMessage` do react-hook-form.

7. `TenantConfigSection` (`src/components/configuracoes/tenant-config-section.tsx`) inclui `max_installments_per_month` no mapeamento API → form (valor padrão 2 quando `null`) e no body do `PATCH /api/v1/tenant-config`.

8. A lógica de validação local **não retroage** sobre propostas existentes — afeta apenas a sessão atual de criação/edição. Propostas já salvas continuam sem revalidação (grandfathering).

9. `npm run build` limpo e `npm run lint` sem warnings após a implementação.

### Função `computeInstallmentsPerMonth` — Algoritmo

[Source: `src/lib/installment-utils.ts`]

`addMonthsClamped` (privada, linha 7) já implementa a expansão de datas com clamp de fim de mês. A nova função pode usá-la internamente.

Pseudocódigo:

```ts
export function computeInstallmentsPerMonth(
  schedules: SaleFormData['installment_schedules']
): Map<string, number> {
  const map = new Map<string, number>()

  const add = (dateStr: string, qty: number) => {
    if (!dateStr) return
    const key = dateStr.slice(0, 7) // 'YYYY-MM'
    map.set(key, (map.get(key) ?? 0) + qty)
  }

  const intervalMap: Record<string, number> = {
    monthly: 1, bimonthly: 2, quarterly: 3, semestral: 6
  }

  for (const s of schedules) {
    if (s.kind === 'entry' || s.recurrence_type === null) {
      // one-shot: specific_date only
      if (s.specific_date) add(s.specific_date, s.quantity ?? 1)
    } else if (s.recurrence_type === 'yearly') {
      if (!s.start_date || !s.recurrence_day) continue
      const start = new Date(s.start_date)
      for (let i = 0; i < (s.quantity ?? 1); i++) {
        const d = new Date(start)
        d.setFullYear(d.getFullYear() + i)
        add(formatDateISO(d.getFullYear(), d.getMonth() + 1, d.getDate()), 1)
      }
    } else {
      // monthly/bimonthly/quarterly/semestral
      const interval = intervalMap[s.recurrence_type] ?? 1
      if (!s.start_date || !s.recurrence_day) continue
      const start = new Date(s.start_date)
      for (let i = 0; i < (s.quantity ?? 1); i++) {
        const d = addMonthsClamped(start, i * interval, s.recurrence_day)
        add(formatDateISO(d.getFullYear(), d.getMonth() + 1, d.getDate()), 1)
      }
    }
  }

  return map
}
```

> **Atenção:** `addMonthsClamped` é função privada (não exportada). Duas opções: exportá-la, ou duplicar a lógica na nova função. Preferir exportar com nome claro.

{violatedMonths.length > 0 && (

### Tenant Config — Campo na API

[Source: `docs/epics/epic-4-nova-proposta-melhorias.md#story-4.5`]

`GET /api/v1/tenant-config` já retorna `max_installments_per_month: number` (int 1–5, default 2).
`PATCH /api/v1/tenant-config` já aceita `max_installments_per_month`.

O tipo gerado pelo `@cacenot/construct-pro-api-client` pode não incluir o campo ainda. Se não incluir, usar cast:

```ts
const cap = (config as { max_installments_per_month?: number }).max_installments_per_month ?? 2
```

E no body do PATCH:
```ts
// max_installments_per_month pode não estar no tipo do client ainda
...(data.max_installments_per_month && {
  max_installments_per_month: data.max_installments_per_month,
}) as Record<string, unknown>
```

### Slider HTML nativo vs. shadcn `<Slider>`

```tsx
<input
  type="range"

  step={1}
  value={field.value}
  onChange={(e) => field.onChange(Number(e.target.value))}
  className="w-full accent-primary"
/>
```

Com display do valor atual ao lado: `<span className="font-medium tabular-nums">{field.value}</span>`.
