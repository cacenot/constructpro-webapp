## Story

**As a** corretor usando o formulário de nova proposta ou edição,
**I want** poder adicionar múltiplas entradas, podendo cada uma ser paga via dinheiro ou via bem (veículo, imóvel, terreno, barco),
**so that** eu possa estruturar propostas com entradas parceladas em bem e refletir fielmente a realidade do negócio para a API `POST /api/v1/sales`.

## Acceptance Criteria

1. O bloco "Entrada" do formulário (`sale-form-step2.tsx`) suporta **N entradas** (mínimo 1, sem limite hard): botão "+ Adicionar Entrada" insere uma nova entrada abaixo das existentes. A primeira entrada não pode ser removida; as demais têm botão de remoção.
2. Cada entrada exibe os campos existentes: **Valor**, **Data de Pagamento** e **Forma de Pagamento** (boleto / pix / dinheiro / transferência / cartão / **bem**).
3. Ao selecionar **"Bem"** como forma de pagamento, o campo **"Tipo de Bem"** aparece (`vehicle | real_estate | land | boat`), com label em português. Trocar o tipo de bem limpa os campos de metadata do sub-form anterior.
4. O sub-form de `asset_proposal` expande condicionalmente de acordo com o tipo selecionado:
   - **vehicle:** plate (string), renavam (string), brand (string), model (string), year (number 1900–2030)
   - **real_estate:** address (string), property_type (string), area_sqm (number > 0), registration_number (string)
   - **land:** address (string), area_sqm (number > 0), registration_number (string)
   - **boat:** registration (string), length_meters (number > 0), brand (string), model (string), year (number 1900–2030)
5. Todos os campos do sub-form de bem são **obrigatórios** quando `payment_method = "asset"` — validação Zod exibe mensagens em português.
6. Entradas com `payment_method ≠ "asset"` não exigem e não exibem campos de asset.
7. O schema Zod (`installmentScheduleSchema`) é atualizado para incluir `payment_method: "asset"` em `paymentMethodValues` e o sub-objeto `asset_proposal` condicional.
8. O resumo financeiro lateral (se existir) ou o subtotal visível reflete a **soma de todas as entradas** em tempo real.
9. As mesmas features estão disponíveis no formulário de **edição** (`sale-edit-form.tsx`): N entradas, seleção de bem, sub-form por tipo.
10. Propostas sem campo `asset` (entradas em dinheiro/pix/boleto) continuam funcionando sem alteração.
11. O payload enviado para `POST /api/v1/sales` inclui, para entradas via bem, o campo `asset_proposal` com a estrutura correta por tipo: `{ type, asset_metadata: { ...campos } }`.

### Estado Atual — Bloco de Entrada

[Source: `src/components/vendas/sale-form-step2.tsx:444–545`]

Atualmente o bloco "Entrada" é **um único campo hardcoded** com `installment_schedules.0.*`. A refatoração transforma isso em array dinâmico de entradas via `useFieldArray`. O `useFieldArray` já existe no componente pai; `sale-form-step2.tsx` recebe `fields`, `append`, `remove` como props.

**Índices reais:** Como `installment_schedules` é um array misto (entry + regular + balloon + etc.), as entradas não são necessariamente os índices 0, 1, 2. A Task 2.2 extrai apenas os `kind="entry"` com seus índices reais no array original.

```ts
// Exemplo de como filtrar entradas com índices reais:
const entryFields = fields
  .map((field, index) => ({ field, index }))
  .filter(({ field }) => field.kind === 'entry')
```

### Schema `asset_proposal` — Contrato com API

[Source: `docs/epics/epic-4-nova-proposta-melhorias.md#story-4.3`]

A API `POST /api/v1/sales` já aceita `payment_method="asset"` + `asset_proposal` em schedules de `kind="entry"`. Estrutura esperada:

```json
{
  "kind": "entry",
  "payment_method": "asset",
  "quantity": 1,
  "amount_cents": 8000000,
  "specific_date": "2026-08-01",
  "asset_proposal": {
    "type": "vehicle",
    "asset_metadata": {
      "plate": "ABC-1234",
      "renavam": "12345678901",
      "brand": "Toyota",
      "model": "Corolla",
      "year": 2022
    }
  }
}
```

**Tipos por `asset_proposal.type`:**

| type | Campos obrigatórios |
|------|---------------------|
| `vehicle` | plate, renavam, brand, model, year |
| `real_estate` | address, property_type, area_sqm, registration_number |
| `land` | address, area_sqm, registration_number |
| `boat` | registration, length_meters, brand, model, year |

### `@cacenot/construct-pro-api-client` — Compatibilidade

[Source: `docs/architecture/vendas-e-contratos.md`]

Se o tipo gerado pelo client não incluir `asset_proposal` no body de `POST /api/v1/sales`, usar cast com comentário:

```ts
// asset_proposal não está no tipo do client ainda — API já aceita
const scheduleWithAsset = schedule as typeof schedule & {
  asset_proposal?: { type: string; asset_metadata: Record<string, unknown> }
}
```

Não usar `as any` puro — preferir cast estruturado.
