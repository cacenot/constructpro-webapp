# UX Spec — Story 2.2: Módulo Corretor (CRUD completo)

## Objetivo da tela / componente

Fornecer ao usuário autenticado (vendedor ou gerente) um módulo completo de gestão de corretores com 4 superfícies:

1. **`/corretores`** — Lista paginada com busca e ações rápidas
2. **`/corretores/novo`** — Formulário de cadastro
3. **`/corretores/:id`** — Página de detalhe
4. **`/corretores/:id/editar`** — Formulário de edição

O módulo segue o **padrão canônico `clientes`** (já validado e em produção), com adaptações específicas para os campos e regras do corretor imobiliário.

---

## Fluxo principal do usuário

### Fluxo 1 — Consultar e agir na lista

```
Sidebar "Corretores"
  → /corretores (lista carrega com skeleton)
  → Tabela renderiza linhas
  → [Opcional] Digitar busca → debounce 300ms → nova query
  → Ação: clicar "⋮" na linha → dropdown "Ações"
    → "Ver detalhes" → /corretores/:id
    → "Editar"       → /corretores/:id/editar
    → "Excluir"      → AlertDialog de confirmação → DELETE → toast → redirect lista
```

### Fluxo 2 — Cadastrar novo corretor

```
/corretores → botão "Novo Corretor"
  → /corretores/novo
  → Preencher formulário (CPF com máscara, nome, CRECI, e-mail, telefone)
  → Submit → loading no botão → POST /api/v1/brokers
  → Sucesso: toast "Corretor cadastrado com sucesso!" + redirect /corretores
  → Erro: toast de erro (handleApiError)
```

### Fluxo 3 — Editar corretor

```
/corretores/:id → botão "Editar"  (ou via dropdown na lista)
  → /corretores/:id/editar
  → Form pré-preenchido com dados do GET /api/v1/brokers/{id}
  → Alterar campos → Submit → PATCH /api/v1/brokers/{id}
  → Sucesso: toast "Corretor atualizado com sucesso!" + redirect /corretores/:id
```

### Fluxo 4 — Excluir corretor (soft-delete)

```
/corretores/:id  OU  dropdown na lista → "Excluir"
  → AlertDialog abre:
      Título: "Excluir corretor?"
      Descrição: "Esta ação removerá o corretor {nome} do sistema. Deseja continuar?"
      Botões: "Cancelar" (outline) | "Excluir" (destructive, vermelho)
  → Confirmar → DELETE /api/v1/brokers/{id}
  → Sucesso: toast "Corretor removido com sucesso!" + redirect /corretores
```

## Estados da UI

### Lista `/corretores`

| Estado | Gatilho | Aparência |
|--------|---------|-----------|
| **Loading inicial** | Primeira carga / mudança de filtro | 10 `<Skeleton>` rows com altura de linha padrão |
| **Dados carregados** | Query resolvida com itens | Tabela com linhas, info "X–Y de Z corretores" |
| **Empty — sem filtro** | Lista vazia, sem busca ativa | Ícone + "Nenhum corretor cadastrado" + botão "Cadastrar primeiro corretor" |
| **Empty — com filtro** | Busca sem resultado | Ícone + "Nenhum corretor encontrado para '{busca}'" + link "Limpar busca" |
| **Erro de rede** | Query falhou | Mensagem de erro + botão "Tentar novamente" (padrão TanStack Query) |

### Formulários (`/novo` e `/editar`)

| Estado | Gatilho | Aparência |
|--------|---------|-----------|
| **Idle** | Página carregada | Campos habilitados, botão "Cadastrar" / "Atualizar" ativo |
| **Loading dados (edição)** | GET pendente no `/editar` | Skeleton covering form fields |
| **Submitting** | Submit clicado | Botão desabilitado + `<Loader2 animate-spin>` + "Salvando..." |
| **Erro de validação** | Zod rejeita | `<FormMessage>` abaixo do campo com texto em vermelho |
| **Erro de API** | `handleApiError` acionado | Toast `error` (sonner, `bottom-center`, `richColors`) |
| **CPF em edição** | Modo `isEdit=true` | Campo CPF desabilitado (não editável — mesmo padrão do cliente PF) |

### Detalhe `/corretores/:id`

| Estado | Gatilho | Aparência |
|--------|---------|-----------|
| **Loading** | GET pendente | Skeleton header + skeleton fields |
| **Dados carregados** | Sucesso | Campos legíveis: CPF mascarado `123.456.789-09`, telefone formatado |
| **Não encontrado** | 404 ou erro | "Corretor não encontrado" + link "Voltar para lista" |

### AlertDialog (exclusão)

| Estado | Gatilho | Aparência |
|--------|---------|-----------|
| **Fechado** | Default | Invisível |
| **Aberto** | Clicar "Excluir" | Dialog modal com título, descrição e dois botões |
| **Confirmando** | Botão "Excluir" clicado | Botão destructive desabilitado + spinner |

## Comportamentos esperados

### Campo CPF

- **Componente:** `<CPFInput>` de `src/components/ui/document-input.tsx` — mesmo do módulo clientes. **Não duplicar.**
- **Máscara visual:** `123.456.789-09` (aplicada em tempo real)
- **Ao submeter:** strip `replace(/\D/g, '')` antes de enviar para API
- **Validação:** `validateCPF` de `@cacenot/construct-pro-api-client`
- **Em edição:** campo desabilitado (`disabled`)

### Campo Telefone

- **Componente:** `<PhoneInput>` de `src/components/ui/phone-input.tsx` — mesmo do módulo clientes.
- **Formato de entrada:** `(11) 98765-4321` com `defaultCountry="BR"` e `international`
- **Ao submeter:** valor já sai em formato E.164 pelo componente (`+5511987654321`)
- **Campo opcional:** `FormLabel` sem asterisco

### Campo CRECI

- **Componente:** `<Input>` padrão shadcn
- **Placeholder:** `"Ex: CRECI-SP 12345"`
- **Validação:** 5–20 caracteres, obrigatório

### Busca na lista

- Debounce: 300ms após última tecla
- Ao modificar busca: reset automático para página 1
- Placeholder: `"Buscar por nome, CPF ou CRECI..."`
- Botão "Limpar filtros" aparece apenas quando `hasActiveFilters === true`

### Paginação

- Mínimo 10 itens por página (`PAGE_SIZE = 10`)
- Info: `"X–Y de Z corretores"` (ex: `"1–10 de 47 corretores"`)
- Botões de página numéricos via `<Pagination>` shadcn

### Tabela — colunas e responsividade

| Coluna | Mobile (`< md`) | Desktop (`≥ md`) | Notas |
|--------|----------------|-----------------|-------|
| Nome completo | ✅ visível | ✅ visível | Linha 1; CPF mascarado na linha 2 (subordinado) |
| CPF | ✅ subordinado ao nome | ✅ subordinado ao nome | `formatDocument()` de `src/lib/text-formatters.ts` |
| CRECI | ✅ visível | ✅ visível | texto pequeno, `text-muted-foreground` |
| E-mail | ❌ `hidden md:table-cell` | ✅ visível | `text-muted-foreground`, truncar com `truncate` |
| Telefone | ❌ `hidden md:table-cell` | ✅ visível | `formatPhone()` de `src/lib/text-formatters.ts` |
| Ações | ✅ visível | ✅ visível | `DropdownMenu` — ver abaixo |

### Dropdown de ações (coluna "Ações")

```
DropdownMenuLabel    → "Ações"
DropdownMenuItem     → "Ver detalhes"   → navigate(/corretores/:id)
DropdownMenuItem     → "Editar"         → navigate(/corretores/:id/editar)
DropdownMenuSeparator
DropdownMenuItem     → "Excluir"        → abre AlertDialog  [classe text-destructive]
```

- Trigger (`<Button variant="ghost" size="icon-sm">`): ícone `MoreVertical`
- Trigger envolto em `<Tooltip>` com label `"Ações"`
- **Sem ícones** nos itens do dropdown (padrão CLAUDE.md)

### Navegação — Sidebar

- Grupo: **"Comercial"** (`navComercial` array em `app-sidebar.tsx`)
- Ícone: **`UserCheck`** (lucide-react) — representa corretor/agente com check de credencial
- Item: `{ title: 'Corretores', href: '/corretores', icon: UserCheck }`
- Link ativo destacado ao acessar qualquer rota `/corretores/*`

### Layout da página de detalhe

Layout simples (sem tabs — corretor não tem sub-entidades no escopo da story 2.2):

```
AppLayout
  └── div.space-y-6
        ├── Header area
        │     ├── Botão "Voltar" (ghost icon ArrowLeft, Tooltip "Voltar")
        │     ├── Nome do corretor (h1)
        │     └── Botões: "Editar" (outline) + "Excluir" (destructive)
        └── Card "Dados do Corretor"
              └── Grid de campos:
                    Nome, CPF (mascarado), CRECI
                    E-mail (se presente), Telefone (formatado, se presente)
```

> **Nota:** página de detalhe do cliente usa tabs (`Visão Geral`, `Contratos`, etc.) por ter sub-entidades. Corretor na story 2.2 **não tem** sub-entidades → layout flat sem tabs. Tabs serão adicionadas no Epic 3 quando o relacionamento corretor × venda existir.
