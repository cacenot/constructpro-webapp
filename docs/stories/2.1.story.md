## História

**Como** desenvolvedor do time frontend,
**Quero** atualizar `@cacenot/construct-pro-api-client` de `0.18.0` para `1.0.0` e corrigir todos os breaking changes revelados pelo type-checker,
**Para que** os novos módulos (Corretor, Imobiliária) e os ajustes de Nova Proposta possam ser implementados contra os contratos corretos da API pós-release PR #40.

## Critérios de Aceitação

1. `package.json` referencia `@cacenot/construct-pro-api-client@1.0.0` e o lock file está atualizado.
2. `npm run build` (TypeScript check + Vite build) finaliza com **zero erros** após o bump.
3. `npm run lint` finaliza com **zero warnings** após o bump.
4. `src/schemas/sale.schema.ts` — `INSTALLMENT_KIND_LABELS` (ou equivalente) reflete os novos `InstallmentKind`: `entry`, `regular`, `balloon`, `key_delivery`, `extra`. Os valores antigos `monthly` e `yearly` **não existem mais** como `kind`.
5. Os novos paths `/api/v1/brokers` e `/api/v1/agencies` estão disponíveis no client tipado (verificado via `useApiClient()` e os tipos gerados).
6. `SaleResponse` possui os campos `commission`, `broker` e `agency` acessíveis nos tipos TypeScript.
7. `SaleCreate` e `SaleUpdate` possuem os campos `broker_id`, `agency_id`, `commission_broker_rate`, `commission_agency_rate` nos tipos TypeScript.
8. Nenhuma regressão visual ou funcional nas telas existentes: autenticação, clientes, projetos, unidades e vendas (Epic 1 — stories 1.x).
9. O PR é **isolado**: não mistura alterações de funcionalidades novas (módulo Corretor/Imobiliária) com o bump.
10. Um comentário no PR documenta os breaking changes encontrados e como foram resolvidos.

### Breaking Change Crítica: InstallmentKind

[Source: DIRECIONAMENTO-FRONTEND.md §0]

**Antes (0.18.0):**
```ts
enum InstallmentKind {
  monthly = 'monthly',
  yearly = 'yearly',
  entry = 'entry',
  extra = 'extra',
}
```

**Depois (1.0.0):**
```ts
enum InstallmentKind {
  entry = 'entry',
  regular = 'regular',      // era monthly/yearly recorrente
  balloon = 'balloon',      // reforço/balão
  key_delivery = 'key_delivery',
  extra = 'extra',
}

enum InstallmentPeriodicity {
  one_shot = 'one_shot',
  monthly = 'monthly',
  bimonthly = 'bimonthly',
  quarterly = 'quarterly',
  semestral = 'semestral',
  yearly = 'yearly',
}
```

**Arquivos do front a verificar:**
- `src/schemas/sale.schema.ts` — `INSTALLMENT_KIND_LABELS`, qualquer validação Zod com `.enum([...kinds...])`
- `src/components/vendas/sale-form.tsx` (~757 linhas) — select de tipo de parcela, lógica de montagem
- `src/components/vendas/sale-edit-form.tsx` — mesmas referências
- `src/lib/installment-utils.ts` — cálculo de datas por tipo

### Novos Campos em SaleCreate / SaleResponse

[Source: DIRECIONAMENTO-FRONTEND.md §2.7]

```ts
// SaleCreate / SaleUpdate (novos campos):
broker_id?: number
agency_id?: number
commission_broker_rate?: number   // Rate em PPM (1% = 10.000)
commission_agency_rate?: number   // Rate em PPM

// SaleResponse (novos campos):
commission?: CommissionResponse   // snapshot imutável criado no approve
broker?: BrokerResponse           // aninhado
agency?: AgencyResponse           // aninhado
```

Esses campos **não precisam ser usados nesta story** — apenas confirmar que os tipos compilam.

[Source: docs/epics/epic-1-vendas-contratos.md — Story 1.4]
