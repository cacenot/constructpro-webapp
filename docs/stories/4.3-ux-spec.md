# UX Spec — Story 4.3: Multi-entrada + entrada via bem (asset)

## Objetivo da tela / componente

Refatorar o bloco **"Entrada"** do Card de Pagamento (`sale-form-step2.tsx` e `sale-edit-form.tsx`) de campo único hardcoded para um sistema dinâmico de N entradas, cada uma podendo ser paga em dinheiro/pix/boleto ou **via bem** (veículo, imóvel, terreno, barco) com sub-form condicional de `asset_proposal`.

**Usuário-alvo:** Corretor que estrutura propostas com entrada parcelada em bem (ex.: cliente dá um carro + R$ 50k de entrada). Precisa registrar fielmente a realidade do negócio sem fricção.

---

## Fluxo principal do usuário

### Cenário A — Entrada simples (sem bem)
1. Corretor preenche formulário de nova proposta → chega no Step 2 (Pagamento)
2. Vê bloco "Entrada" com 1 entrada já preenchida (pré-existente)
3. Preenche Valor, Data de Pagamento, Forma de Pagamento (pix/boleto/etc.)
4. Segue para Parcelas normalmente — sem mudança de comportamento

### Cenário B — Múltiplas entradas
1. Corretor clica em **"+ Adicionar Entrada"** abaixo do bloco de entradas
2. Nova entrada aparece abaixo da primeira com todos os campos em branco
3. Botão de remoção `Trash2` aparece no canto direito da 2ª entrada (não na 1ª)
4. Corretor preenche a nova entrada
5. Pode adicionar quantas entradas quiser repetindo o processo

### Cenário C — Entrada via bem
1. Na entrada desejada, corretor seleciona **"Bem"** em Forma de Pagamento
2. Sub-form expande abaixo dos campos principais da entrada
3. `<Select>` "Tipo de Bem" aparece (Veículo / Imóvel / Terreno / Barco)
4. Corretor seleciona o tipo → campos específicos do tipo aparecem
5. Corretor preenche todos os campos obrigatórios
6. Validação em tempo real mostra erros se algum campo obrigatório estiver vazio

### Cenário D — Edição de proposta existente com asset
1. Corretor abre proposta que já tem entrada via bem
2. Sub-form carrega pré-populado com os dados do `asset_proposal` existente
3. Corretor pode alterar o tipo de bem → campos do tipo anterior são limpos
4. Pode trocar para forma de pagamento normal → sub-form colapsa

## Layout do bloco de entradas

### Estrutura visual

```
┌─ Card "Pagamento" ─────────────────────────────────────────────┐
│                                                                  │
│  ── Entrada ──────────────────────────────────────── [+ Entrada]│
│                                                                  │
│  ┌─ Entrada 1 ───────────────────────────────────────────────┐  │
│  │  [Valor *]      [Data de Pagamento *]  [Forma Pagamento *] │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌─ Entrada 2 ─────────────────────────────────── [🗑 Remover]┐  │
│  │  [Valor *]      [Data de Pagamento *]  [Bem ▼]             │  │
│  │  ─────────────────────────────────────────────────────── │  │
│  │  [Tipo de Bem: Veículo ▼                               ]   │  │
│  │  [Placa *]    [RENAVAM *]    [Marca *]                     │  │
│  │  [Modelo *]                 [Ano *   ]                     │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ── Separator ────────────────────────────────────────────────  │
│                                                                  │
│  ── Parcelas ─────────────────────────────── [+ Adicionar ▼]   │
│  ...                                                             │
└──────────────────────────────────────────────────────────────────┘
```

### Decisão de layout: entrada por card individual

**Padrão:** Cada entrada é uma `div` com `border border-border/50 rounded-xl p-4 bg-card` — mesmo estilo do design system (sem usar `<Card>` completo para não criar depth excessiva dentro do Card pai).

Não usar lista flat: entradas com sub-form podem ter alturas muito diferentes, o que tornaria a lista visualmente confusa. Card individual delimita claramente o escopo de cada entrada.

### Header da seção de entradas

```
linha horizontal: "Entrada" (text-sm font-medium) + botão "+ Adicionar Entrada" (à direita)
```

O botão **"+ Adicionar Entrada"** fica no header da seção (alinhado à direita), não abaixo das entradas, seguindo o padrão do botão "Adicionar Parcelas" existente.

```tsx
// Padrão
<div className="mb-3 flex items-center justify-between">
  <p className="text-sm font-medium">Entrada</p>
  <Tooltip>
    <TooltipTrigger asChild>
      <Button type="button" variant="outline" size="sm">
        <Plus className="mr-2 size-4" />
        Adicionar Entrada
      </Button>
    </TooltipTrigger>
    <TooltipContent>
      <p>Adicionar nova entrada</p>
    </TooltipContent>
  </Tooltip>
</div>
```

### Botão de remoção por entrada

- Posição: `absolute top-3 right-3` dentro do card da entrada (usar `relative` no container)
- Componente: `Button variant="ghost" size="icon"` com ícone `Trash2 size-4`
- Visível: apenas em entradas 2+ (índice real > índice da primeira entrada `kind="entry"`)
- Tooltip: "Remover entrada"
- Cor do ícone: `text-muted-foreground hover:text-destructive`

```tsx
// Estrutura do card de entrada
<div className="relative rounded-xl border border-border/50 p-4 space-y-4">
  {isNotFirst && (
    <div className="absolute right-3 top-3">
      <Tooltip>
        <TooltipTrigger asChild>
          <Button
            type="button"
            variant="ghost"
            size="icon"
            onClick={() => remove(realIndex)}
            className="size-7 text-muted-foreground hover:text-destructive"
          >
            <Trash2 className="size-4" />
          </Button>
        </TooltipTrigger>
        <TooltipContent>
          <p>Remover entrada</p>
        </TooltipContent>
      </Tooltip>
    </div>
  )}

  {/* Campos principais */}
  <div className="grid gap-4 sm:grid-cols-12">
    {/* Valor (col-span-4), Data (col-span-4), Forma de Pagamento (col-span-4) */}
    {/* Se isNotFirst: Valor e Data cada um com pr-10 para não sobrepor o botão de remoção */}
  </div>

  {/* Sub-form de bem — condicional */}
  {paymentMethod === 'asset' && <AssetSubForm ... />}
</div>
```

## Sub-form de bem (asset_proposal)

### Aparência e expansão

O sub-form expande **diretamente abaixo** dos campos principais da entrada, separado por `<Separator />` com `className="my-1"`. Sem animação de accordion — aparece/desaparece diretamente (sem frills, forma segura em react-hook-form).

### Seletor de tipo de bem

```
[Tipo de Bem ▼                              ]   (full width, sm:col-span-12)
```

Label: "Tipo de Bem *" | Opções: Veículo / Imóvel / Terreno / Barco

Ao trocar o tipo: campos do tipo anterior somem, campos do novo tipo aparecem. `asset_proposal.asset_metadata` é resetado via `form.setValue`.

### Grids de campos por tipo

**Todos os campos marcados com `*` no label** (campos obrigatórios quando `payment_method = "asset"`).

#### Veículo (`vehicle`)

```
Linha 1: [Placa * — col-4] [RENAVAM * — col-4] [Marca * — col-4]
Linha 2: [Modelo * — col-8]                    [Ano * — col-4]
```

#### Imóvel (`real_estate`)

```
Linha 1: [Endereço * — col-8]         [Tipo de Imóvel * — col-4]
Linha 2: [Área (m²) * — col-6]        [Nº de Registro * — col-6]
```

#### Terreno (`land`)

```
Linha 1: [Endereço * — col-8]         [Área (m²) * — col-4]
Linha 2: [Nº de Registro * — col-12]
```

#### Barco (`boat`)

```
Linha 1: [Registro * — col-4]  [Comprimento (m) * — col-4]  [Marca * — col-4]
Linha 2: [Modelo * — col-8]                                  [Ano * — col-4]
```

### Campos numéricos

- `year` e `area_sqm` e `length_meters`: `<Input type="number" />` com `className="tabular-nums"` (padrão do design system para valores numéricos)
- `year`: `placeholder="2024"`, `min={1900}`, `max={2030}`
- `area_sqm`, `length_meters`: `placeholder="0"`, `min={0}` (validação positivo via Zod)

### Componente de campo padrão

```tsx
// Padrão FormField → FormItem → FormLabel + FormControl + FormMessage
<FormField
  control={form.control}
  name={`installment_schedules.${realIndex}.asset_proposal.asset_metadata.plate`}
  render={({ field }) => (
    <FormItem className="sm:col-span-4">
      <FormLabel>Placa *</FormLabel>
      <FormControl>
        <Input placeholder="ABC-1234" {...field} />
      </FormControl>
      <FormMessage />
    </FormItem>
  )}
/>
```

## Estados da UI

### Estado 1 — Default (1 entrada, sem bem)

- 1 card de entrada com os 3 campos (Valor, Data, Forma de Pagamento)
- Sem botão de remoção
- Sem sub-form
- Botão "+ Adicionar Entrada" habilitado no header

### Estado 2 — Múltiplas entradas

- N cards de entrada empilhados verticalmente com `space-y-3`
- Entrada 1: sem botão de remoção
- Entradas 2+: botão `Trash2` no canto superior direito
- Cada entrada independente (valor, data, forma de pagamento)

### Estado 3 — Entrada com bem selecionado (sem tipo)

- Campos principais visíveis
- `<Separator />` aparece
- `<Select>` "Tipo de Bem *" aparece vazio com placeholder "Selecione o tipo"
- Campos do asset ainda não visíveis (aguardando seleção de tipo)
- Se tentar submeter sem tipo: `FormMessage` abaixo do select

### Estado 4 — Tipo de bem selecionado (campos visíveis)

- Select de "Tipo de Bem" com valor selecionado
- Grid de campos do tipo selecionado abaixo
- Campos vazio com placeholder em cada um

### Estado 5 — Validação com erro

- Campo com erro: borda vermelha automática via shadcn (controlled by `FormMessage`)
- `<FormMessage />` exibe mensagem Zod em PT-BR abaixo de cada campo com erro
- Exemplo: "Placa é obrigatória", "Ano deve estar entre 1900 e 2030"
- **Não bloquear input** — validação on-blur / on-submit, não on-change

### Estado 6 — Asset pré-populado (edit form)

- Sub-form aparece já expandido com os dados carregados do `asset_proposal`
- Tipo de bem já selecionado no select
- Todos os campos do tipo preenchidos com os valores existentes
- Corretor pode editar qualquer campo

### Estado 7 — Trocar tipo de bem

- Corretor muda o select de tipo (ex.: Veículo → Imóvel)
- Campos do Veículo desaparecem imediatamente
- Campos do Imóvel aparecem em branco (dados anteriores limpos)
- Sub-form de Tipo de Bem permanece com o novo tipo selecionado

### Estado 8 — Trocar forma de pagamento (asset → pix)

- Sub-form de bem colapsa completamente (sem `<Separator />`)
- `asset_proposal` é setado para `null` no form
- Campos do bem não aparecem mais para esta entrada

## Comportamentos esperados

### Adição de entrada

- Clique em "+ Adicionar Entrada" → `append({ kind: 'entry', payment_method: 'pix', quantity: 1, amount: 0, specific_date: null, asset_proposal: null })`
- Nova entrada aparece abaixo de todas as entradas existentes
- Scroll automático para a nova entrada: `entry.scrollIntoView({ behavior: 'smooth', block: 'nearest' })`
- Foco no campo "Valor" da nova entrada

### Remoção de entrada

- Clique no `Trash2` → `remove(realIndex)` (índice real no array `installment_schedules`)
- Sem modal de confirmação — ação imediata (valores monetários pequenos, UX fluida prioritária)
- Ao remover, entradas seguintes se renumeradas automaticamente pelo re-render

### Seleção "Bem" em Forma de Pagamento

- `onValueChange` do select de forma de pagamento
- Se novo valor = `'asset'`: setar `asset_proposal: { type: undefined, asset_metadata: {} }` via `form.setValue`
- Sub-form aparece: primeiro `<Select>` de tipo de bem, campos ainda ocultos

### Troca de tipo de bem

- `onValueChange` do select de tipo de bem
- `form.setValue(`installment_schedules.${realIndex}.asset_proposal.asset_metadata`, getDefaultMetadata(newType))`
- `getDefaultMetadata`: retorna objeto com os campos do tipo em string/number vazios (não `undefined`) para evitar campos não controlados:

```ts
const getDefaultAssetMetadata = (type: AssetType) => {
  switch (type) {
    case 'vehicle': return { type, plate: '', renavam: '', brand: '', model: '', year: '' as unknown as number }
    case 'real_estate': return { type, address: '', property_type: '', area_sqm: '' as unknown as number, registration_number: '' }
    case 'land': return { type, address: '', area_sqm: '' as unknown as number, registration_number: '' }
    case 'boat': return { type, registration: '', length_meters: '' as unknown as number, brand: '', model: '', year: '' as unknown as number }
  }
}
```

### Troca de forma de pagamento (asset → outro)

- `form.setValue(`installment_schedules.${realIndex}.asset_proposal`, null)`
- Sub-form some completamente

### Validação visual em tempo real

- Erros aparecem ao sair de campo (on-blur) via react-hook-form mode
- FormMessage persiste até o campo ser corrigido
- Submit bloqueado enquanto houver erros (react-hook-form padrão)

### Acessibilidade

- Todos os labels associados via `htmlFor` (FormLabel + FormControl padrão shadcn)
- Botão Trash2: `aria-label="Remover entrada {n}"` via Tooltip (shadcn já lida com isso)
- Select de tipo de bem: `aria-required="true"` quando `payment_method === 'asset'`
- Ordem de foco natural (top-to-bottom, left-to-right) — sem manipulação de tabIndex

## Notas para @dev

1. **Não criar `<AssetSubForm>` como componente separado** se não existir padrão similar no projeto — manter como seção JSX inline no render de cada entrada (KISS)
2. **`payment_method: 'asset'` no select** usa o mesmo `PAYMENT_METHOD_LABELS` que os outros métodos — adicionar a key `asset: 'Bem'` a este objeto
3. **Numeração das entradas** ("Entrada 1", "Entrada 2"): exibir apenas se houver 2+ entradas para não poluir o layout com 1 entrada
4. **Edit form**: `sale-edit-form.tsx` carrega dados via `useFieldArray` — ao popular, o `asset_proposal` já existente no schedule de tipo `entry` popula o sub-form automaticamente se o watch/register estiver correto
5. **Scroll na adição**: `useEffect` + `ref` no último card de entrada para scroll suave após `append`
